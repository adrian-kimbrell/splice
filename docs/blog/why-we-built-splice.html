Why we built Splice

April 14, 2026

This started with a frustration. I was running Claude Code on three projects at once — a frontend, a backend, and a side experiment — each in its own terminal window. Every time an agent hit a permission prompt or needed a decision, I'd have to find the right window, read the context, respond, and then remember what I was doing before.

That's not a workflow problem. That's a tooling problem.

The missing abstraction

Terminal emulators haven't changed much. They give you a window, maybe tabs, maybe a split. That was the right model when you were the one running commands. But when you're running agents — processes that work autonomously for minutes or hours at a time — the window-per-project model breaks down.

You need to see everything at once. You need to know when something needs you. You need to be able to jump into any context instantly, without losing what you were doing somewhere else.

None of the tools I was using were built for this. iTerm2 is a great terminal emulator. VS Code is a great editor. But stringing them together for multi-agent, multi-project work means a lot of manual state management that gets in the way of actually building things.

Workspaces as the core primitive

The first thing we built was workspaces. Not tabs, not windows — workspaces. Each one is a fully isolated environment: its own project folder, its own terminals, its own open files, its own layout. Switching between them is a keystroke. Everything is exactly where you left it.

This sounds simple. In practice, it means:

  • You can have Claude Code running on your backend in one workspace while you write frontend code in another
  • You can keep a long-running build alive in one workspace while you're in a completely different project
  • When you come back to a workspace, your terminals are still running — no reconnect, no loss of context

And when you relaunch Splice, everything comes back. Workspaces, layouts, open files, terminal directories — all restored automatically.

Attention without interruption

The other thing we built is what we call attention hooks. Splice watches for agent activity across all your terminal sessions and surfaces it when something needs you — a permission prompt, an error, a decision point. You see it without having to actively monitor every terminal.

When nothing needs your attention, Splice gets out of the way. The agents work. You work. You only interact when the work requires it.

The terminal itself

We also wrote our own terminal emulator from scratch — Snakebite. Not because existing solutions are bad, but because we needed something we could own completely: rendering, protocol handling, integration with the workspace system.

Snakebite renders directly to a canvas. No DOM per character, no xterm.js overhead. The parsing happens in Rust using vte, which handles the full ANSI escape sequence spec including SGR sub-parameters for underline styles. Scrollback is a ring buffer in Rust — the frontend requests viewport slices, not the whole buffer.

The result is a terminal that feels fast because it is fast. It stays smooth at 120fps even when build output is flooding the screen.

What's next

Splice is early. We have two public releases. There's a lot still to build — auto-update, notarization, Linux and Windows support, more AI workflow tooling.

But the core is solid. If you're working with AI coding agents and you're tired of juggling windows, give it a try. It's free, open source, and MIT licensed.
