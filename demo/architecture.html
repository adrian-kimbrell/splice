Splice Architecture Reference

What is Splice?

Splice is a native desktop code editor — built with Rust + Tauri for the backend and Svelte 5 + TypeScript for the UI. It runs as a native macOS app: no browser chrome, no Electron. The frontend is a webview that talks to the Rust backend over Tauri's IPC bridge.

What does it do?

Multi-workspace code editing with split panes, a canvas-rendered terminal, Language Server Protocol (LSP) support for completions and diagnostics, SSH remote workspaces, file watching, and deep Claude Code integration via an attention hook system.

Key Data Flows

Step-by-step traces of the most important operations in Splice — from user action to result.

⬛
Terminal Rendering Pipeline

How shell output becomes pixels on screen. This loop runs at up to ~120 fps. The entire path from bytes to canvas is under 10ms on modern hardware.

1
Shell process (PTY)
Writes ANSI escape sequences and UTF-8 text bytes to the PTY master. For example: ESC[32mHello\ESC[0m means "print Hello in green then reset colour."
2
Rust — term.rs (VTE parser)
A VTE parser calls GridPerformer for each character and escape sequence. It prints characters into the cell grid, moves the cursor, applies foreground/background colours, and handles bold/italic/underline/inverse flags.
3
Rust — emitter.rs (rate limiter)
A background thread wakes every ~8ms. If the grid changed, it serialises it to a compact binary frame and base64-encodes it.
Header (20 bytes): cols, rows, cursor position, cursor style, mode flags (bracketed paste, mouse mode, alt screen…), scrollback length, first visible row index Cells (12 bytes each, row-major): codepoint (u32 LE), fg RGB, bg RGB, flags, char width
4
Tauri event → terminal:grid:<id>
The base64 frame is emitted as a Tauri event targeted to the specific terminal ID. The webview receives it as a string.
5
renderer.ts — dirty-rect diff
Decodes base64 → Uint8Array. Parses the header. Compares each cell to the previous frame. Only repaints cells that actually changed — a huge performance win for typical typing where most of the screen is static.
6
CanvasTerminal.svelte — canvas 2D
Renders cells using a colour string cache (avoids repeated "rgb(r,g,b)" allocations) and an ASCII character cache (pre-computed strings for codepoints 32–127). Handles cursor blink, text selection overlay, search highlights, scrollbar.
💾
File Save (⌘S)

What happens when you hit ⌘S. For SSH workspaces the IPC command changes but everything else is identical.

1
App.svelte — global keydown handler
⌘S fires the "save" action. Calls workspaceManager.saveActiveFile().
2
workspace-file-ops.ts
Looks up: active workspace → active pane → active file path → OpenFile object. Reads file.content (kept in sync by CodeMirror on every keystroke). Marks file.dirty = false optimistically.
3
IPC: invoke("write_file") or invoke("sftp_write_file")
For local workspaces: calls write_file(path, content). For SSH workspaces: calls sftp_write_file(workspaceId, path, content) instead.
4
Rust — fs/write.rs
Validates path is within allowed roots. Writes to disk. The file watcher will fire an fs:change event — the frontend debounces this and ignores it if the content matches what was just saved.
◈
LSP: Diagnostics & Completions

How error squiggles and autocomplete suggestions reach the editor. LSP servers are started lazily — the first time you open a file of a given language.

1
CodeMirrorEditor.svelte — file open
Calls lspClient.ensureReady(filePath, content, workspaceRoot). The client maps the file extension to a language ID (e.g. .ts → "typescript").
2
lspClient — lazy startup + dedup
If this language isn't running yet, calls invoke("lsp_start"). A startPromises Map ensures that if two files open simultaneously, only one start call is made. A failedLanguages Set prevents retry storms.
3
Rust — lsp/mod.rs
Spawns typescript-language-server --stdio (or equivalent). A writer task drains an mpsc channel into stdin. A reader task parses stdout JSON-RPC frames and routes them:
Response (has id, no method) → wake the waiting invoke() caller Notification (no id) → emit as Tauri event (e.g. lsp:diagnostics) Server request (has both) → answer inline (e.g. workspace/configuration)
4
Tauri event → lsp:diagnostics
Frontend receives the diagnostics push. diagnosticsStore is updated. CodeMirror's linter extension re-renders error/warning squiggles.
5
lsp-extensions.ts — CodeMirror extensions
For completions, buildLspCompletionSource(filePath) fires textDocument/completion via lspClient.complete() and returns results to CodeMirror's autocomplete. For hover, buildHoverExtension() shows type info in a tooltip.
⚡
Claude Attention Hook

How Claude Code signals Splice when it needs permission or finishes a task. The key insight: Claude reads SPLICE_TERMINAL_ID from its environment — no PID tree walking needed.

1
Rust — on app startup
Generates or loads a 128-bit auth token. Binds a random TCP port and writes it to ~/.config/Splice/.attention_port. Installs a Python hook script into ~/.claude/settings.json for the Notification and SessionStart hook types.
2
Rust — PTY spawn injects env var
When spawning any terminal, Rust adds SPLICE_TERMINAL_ID=<n> to the shell environment. This propagates to every child process, including Claude.
3
Claude Code — hook fires
The installed Python script runs. It reads SPLICE_TERMINAL_ID from its env and reads the port from the file. It POSTs to Splice's local HTTP server with a Bearer token:
POST http://127.0.0.1:<port>/attention Authorization: Bearer <token> { "terminal_id": 3, "type": "permission" } ← needs user approval { "terminal_id": 3, "type": "idle" } ← finished task
4
Tauri event → attention:notify
Rust validates the token, routes the payload to the frontend via Tauri event.
5
attentionStore + Terminal tab UI
The notification is stored keyed by terminal ID. The terminal's tab gets a coloured badge. Permission requests show inline allow/deny buttons. Idle notifications show a "done" indicator.
↺
Session Restore on Relaunch

How Splice picks up exactly where you left off — same layout, same files, same terminals, and Claude sessions resumed automatically.

1
App.svelte — onMount
Calls workspaceManager.restoreWorkspaceImpl().
2
IPC: invoke("get_workspaces")
Rust reads the per-window JSON file: workspaces.json for the main window, workspaces-{label}.json for secondary windows. Returns serialised workspace state.
3
workspace-session.ts — deserialise
Remaps snake_case → camelCase. Reconstructs the layout binary tree, pane configs, open files list, and active pane. Rebuilds the openFileIndex lookup map.
4
IPC: spawn_terminal + read_file per item
Respawns each saved terminal. Reads each saved file from disk (detecting external edits that happened while Splice was closed).
5
workspace-session.ts — waitForShellReady()
Polls the terminal frame stream until the cursor position is stable for 3 consecutive frames — indicating the shell prompt has appeared. Then sends claude --resume <session_id> to reattach the previous Claude session.
6
UI renders
Full layout restored: pinned tabs, split panes, expanded folders, active pane, and Claude sessions resuming in their terminals.
⬡
SSH Remote Workspace

Editing files and running a terminal on a remote machine. From the user's perspective it looks identical to a local workspace.

1
SshConnectForm.svelte
User enters host, user, port, key path, remote path. Calls invoke("ssh_connect", { workspaceId, config }).
2
Rust — ssh.rs: ControlMaster session
Uses the openssh crate to open a persistent multiplexed SSH connection. Stored in AppState::ssh_sessions[workspace_id]. All subsequent SFTP operations reuse this one TCP connection.
3
Rust — SFTP for file operations
sftp_list_dir runs a remote find command. sftp_read_file and sftp_write_file transfer complete file contents. The frontend can't tell the difference — it uses the same IPC commands.
4
Rust — Terminal via SSH binary
spawn_terminal is called with extra_args built from the SSH config. The shell executable is /usr/bin/ssh (on the allowed list). The terminal's PTY and canvas renderer work exactly the same — the shell just happens to be remote.
extra_args: ["-i", keyPath, "-p", port, "-t", "-o", "StrictHostKeyChecking=accept-new", "user@host", "cd '/remote/path' && exec $SHELL -l"]
